# Manage TrueNAS with OpenTofu and API app automation

TrueNAS SCALE is managed as a storage-first appliance, not as another Ansible-owned Linux host. We use OpenTofu for low-blast-radius provider-managed configuration such as datasets and AWS backup infrastructure; typed reconcilers own catalog apps, periodic snapshots, NFS exports, and API-key publication where current providers cannot preserve the required boundaries. Stable TrueNAS catalog apps are preferred over custom Compose. Terraform-compatible state lives in an S3 backend owned by a dedicated bootstrap stack, while restic backups use a separate cost-optimized S3 bucket.

The Phase 7 provider review retained those typed boundaries. The broad community provider targets TrueNAS 25.10+ rather than the current 25.04.2.6 appliance, forces catalog-app replacement for version changes, and stores generated API-key values in state. No iXsystems-supported provider currently covers catalog apps, periodic snapshots, users, and API keys without those gaps. Re-evaluate after an appliance upgrade and provider acceptance testing; do not migrate secrets into state merely to reduce custom code.
